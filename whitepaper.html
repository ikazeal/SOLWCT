1. 项目概述

WCT 是围绕赛事预测与社区互动构建的代币化产品。我们通过“预测 + 支持率展示 + 排行榜 + 奖励机制”的组合，为社区提供可视化、可参与、可复盘的赛季玩法。

2. 术语与定义

  • WCT：WorldCup Token，部署在 BNB Smart Chain 的代币。
  • 预测：钱包地址对单场比赛提交主胜 / 平局 / 客胜三选一的结果判断。
  • 支持率：某场比赛某个选项的预测人数占该场总预测人数的比例。
  • 胜率（排行榜）：在已结算比赛中，预测正确的场次 / 已预测且已结算的场次。
  • 奖池（BNB）：用于活动奖励的 BNB 资金池。页面展示口径为“奖池地址的链上 BNB 余额”。

3. 产品形态与数据来源

3.1 当前实现形态

  • 前端站点：用于展示赛程、预测交互、排行榜与奖励信息。
  • 存储模式：可运行于“本地演示存储（浏览器 localStorage）”或“后端存储（API + 服务器持久化）”。具体以部署配置为准。
  • 链上读取：页面会通过 BSC RPC 读取合约与奖池地址数据（例如奖池地址 BNB 余额）。

3.2 赛程与比赛结果来源

  • 赛程优先读取站点内的 schedule.json。
  • 比赛结果可从第三方公开页面抓取并解析（存在结构变动导致失效的风险）。如数据源变更，以站点公告或更新后的实现为准。

4. 预测机制

  • 提交方式：连接钱包后，对每场比赛选择主胜 / 平局 / 客胜并提交。
  • 提交限制：同一钱包同一场比赛通常仅可提交一次（以页面实现与活动公告为准）。
  • 不涉及下注/积分：预测不需要支付积分或下注金额。
  • 每日次数：持有 < 100,000 WCT 的钱包每日最多提交 1 次预测；持有 ≥ 100,000 WCT 的钱包每日最多提交 5 次预测。
  • 持币奖励加成：用于活动奖励结算时的加成系数（不改变比赛结果）。

5. 预测规则与支持率

  • 封盘规则：比赛开始前 5 分钟 停止提交预测。
  • 提交限制：同一钱包同一场比赛通常仅可提交一次（以页面实现与活动公告为准）。
  • 支持率展示：页面展示各选项的预测人数与支持率，用于反映社区倾向。

6. 结算与分配

比赛结束后，系统根据公开赛果来源生成该场比赛的最终结果，并对已提交预测进行校验：

  • 命中：预测选项与最终结果一致。
  • 未命中：预测选项与最终结果不一致。

7. 排行榜口径（胜率）

  • 胜率：预测正确场次 / 已预测且已结算场次。
  • 有效场次门槛：至少完成 20 场“已结算比赛”的预测，才进入排行榜。
  • 综合评分：为兼顾胜率与样本量，排行榜使用综合评分排序：Score = 胜率 × ln(1 + 有效场次)。
  • 同分排序：当综合评分相同，可按“胜率更高 / 正确场次更多 / 更早提交”等规则排序（以活动公告为准）。

1. Overview

WCT is a tokenized product built around match predictions and community participation. The system combines predictions, community support-rate display, leaderboards, and campaign rewards to create a season-style community experience.

2. Definitions

  • WCT: WorldCup Token deployed on BNB Smart Chain.
  • Prediction: a wallet address submits one of Home / Draw / Away as the expected outcome for a match.
  • Support rate: the share of predictions choosing an option for a match.
  • Win rate (leaderboard): correct predictions / scored predictions (predicted & settled matches).
  • Prize pool (BNB): BNB funds intended for campaign rewards. The website displays the on-chain BNB balance of a designated prize pool address.

3. Product Architecture & Data Sources

3.1 Implementation Modes

  • Front-end: displays fixtures, prediction UI, leaderboard, and rewards.
  • Storage: may run in a local demo mode (browser localStorage) or a server-backed mode (API + persistent storage), depending on deployment configuration.
  • On-chain reads: the website reads on-chain data via public BSC RPC endpoints (e.g., prize pool BNB balance).

3.2 Fixtures & Results

  • Fixtures are loaded from schedule.json when available.
  • Match results may be parsed from third-party public sources, which can change without notice and may cause settlement discrepancies.

4. Prediction Mechanism

  • Submission: after connecting a wallet, users submit one of Home / Draw / Away for a match.
  • Limit: typically one prediction per wallet per match (subject to the website implementation and campaign announcements).
  • No staking: predictions do not require staking points or any wager amount.
  • Daily quota: wallets holding < 100,000 WCT can submit up to 1 prediction per day; wallets holding ≥ 100,000 WCT can submit up to 5 predictions per day.
  • Holding reward boost: an optional multiplier applied when calculating campaign rewards (does not change match outcomes).

5. Prediction Rules

  • Close time: predictions close 5 minutes before kickoff.
  • Limit: typically one prediction per wallet per match (subject to the website implementation and campaign announcements).
  • Support rate: the website displays counts and support rates for each option to reflect community sentiment.

6. Settlement & Distribution

After a match ends, the system derives a final result from public result sources and scores predictions:

  • Correct: the submitted option matches the final result.
  • Incorrect: the submitted option does not match the final result.

7. Leaderboard (Win Rate)

  • Win rate: correct predictions / scored predictions (predicted & settled matches).
  • Minimum matches: a wallet must have at least 20 scored predictions to appear on the leaderboard.
  • Composite score: to balance win rate and sample size, the leaderboard ranks by Score = winRate × ln(1 + matches).
  • Tie-breakers: if scores tie, ranking may use higher win rate, more correct picks, or earlier submissions (per campaign announcements).

8. Prize Pool Display & Winner Mechanism

8.1 On-chain prize pool display

  • Prize pool address: 0xfed911c7463e0cd88192d3ae74b8e7dc77504744
  • Display logic: the website displays the on-chain BNB balance of this address (read via public BSC RPC).
  • Important: the displayed balance is not a promise of payout. Actual distribution is determined by each campaign’s published rules.

8.2 Winner mechanism (Top 5)

Each campaign period forms a “total prize pool” used for leaderboard rewards and ecosystem growth. The leaderboard is ranked by the composite score (with minimum-match requirements and tie-breakers), and the top 5 wallets receive prize pool rewards.

  • Rank #1: 40% of the total prize pool
  • Rank #2: 20% of the total prize pool
  • Rank #3: 10% of the total prize pool
  • Rank #4: 5% of the total prize pool
  • Rank #5: 5% of the total prize pool

Remaining 20%: used for marketing campaigns and per-match extra rewards. Timing, eligibility, and exact rules will be announced per campaign.
